# G088 Syncratic Context Windows And Insight Synthesis

Status: implemented â€” backend/API selective materialization validated
Use case: MarketOps Daily Market Surveillance

## Goal

Introduce Syncratic as the explainability layer that builds durable context windows and synthesized insights from existing SignalOps and MarketOps ledgers.

G088 should create the first deterministic substrate for multi-event insights. It should not add a separate Syncratic ingestion layer, call external data sources, generate LLM narratives, mutate alert lifecycle state, deploy policies, or write graph state.

## Core Product Boundary

The system should distinguish alerts and insights by operator intent:

- Alerts are event-level operational work items.
- Insights are multi-event analytical explanations produced from a bounded evidence context.

Current behavior creates one insight per valid signal. That makes alerts and insights feel duplicated because both are derived from the same signal row. G088 should change the target design so insights are generated by a Syncratic context builder over multiple supporting records.

## Syncratic Role

Syncratic is the explanation fabric over SignalOps and MarketOps.

It should answer:

- why this matters;
- which evidence supports it;
- what context window was used;
- what baseline or prior behavior it was compared against;
- which graph proposals or operator decisions are related;
- whether a human accepted, rejected, deferred, or superseded related claims;
- whether the explanation can be reproduced from persisted evidence.

Syncratic should not be a detector or source adapter in this gate. It should consume existing ledgers and persist derived context/explanation records.

## Starting Point

Existing durable evidence sources include:

- `normalized_event_ledger`
- `signal_ledger`
- `alert_ledger`
- `insight_ledger`
- `marketops_dsm_artifacts`
- `marketops_dsm_graph_proposals`
- `marketops_backtest_*` ledgers
- `marketops_backtest_evaluation_labels`
- `marketops_backtest_evaluations`
- `marketops_backtest_promotion_candidates`

G088 should use these internal ledgers first.

Do not introduce external Syncratic ingestion for:

- news;
- filings;
- analyst notes;
- earnings calendars;
- macro data;
- social feeds;
- third-party taxonomies.

Those can be future gates after internal explainability is working.

## Problem

The current alert/insight lifecycle blurs two concepts:

- alert: an event-level action item;
- insight: a pattern-level explanation.

Because every valid signal can create an insight, the UI may show the same event under both Alerts and Insights. That duplication weakens the meaning of insights and makes it harder for operators to understand what Syncratic contributes.

## Scope

Implement or specify deterministic Syncratic context windows and multi-event insight synthesis.

In scope:

- define context window records;
- define synthesized insight records or adapt existing insight semantics carefully;
- build context windows from existing internal ledgers;
- group supporting alerts/signals/events by deterministic rules;
- persist supporting evidence references;
- expose read/create/list/detail APIs for context windows and synthesized insights;
- document how this changes the alert/insight product boundary;
- keep event-level alert behavior unchanged.

Out of scope:

- separate Syncratic ingestion layer;
- external data connectors;
- LLM-generated explanations;
- automatic graph writes;
- alert lifecycle mutation changes;
- deployment planning or policy execution;
- detector threshold edits;
- ML training;
- PnL/trading simulation.

## Target Flow

Current flow:

```text
event -> signal -> alert + insight -> artifact -> graph proposal -> decision -> evaluation
```

Target G088 flow:

```text
event -> signal -> alert
                  -> Syncratic context builder
                  -> context window
                  -> synthesized insight
                  -> graph proposal / decision / evaluation references
```

Alerts remain close to individual actionable events. Syncratic insights explain patterns across one bounded evidence context.

## Context Window Definition

A context window is the bounded evidence set Syncratic uses to produce or explain an insight.

Minimum fields:

- `context_window_id`
- `tenant_id`
- `app_id`
- `domain`
- `use_case`
- `subject_type`
- `subject_id`
- `subject_symbol`
- `window_start`
- `window_end`
- `context_strategy`
- `context_builder_version`
- `signal_types`
- `detector_ids`
- `event_ids`
- `signal_ids`
- `alert_ids`
- `artifact_ids`
- `graph_proposal_ids`
- `label_ids`
- `baseline_refs`
- `evaluation_refs`
- `promotion_candidate_refs`
- `summary_metrics`
- `evidence_digest`
- `created_at`

The context window should be reproducible. The same strategy, builder version, subject, time range, and evidence query should produce the same deterministic `context_window_id` and evidence digest.

- `tenant_id`

- `use_case`
- `context_window_id`
- `insight_type`

- `subject_symbol`
- `status`

- `title`
- `summary`

- `supporting_alert_ids`
- `supporting_signal_ids`
- `supporting_event_ids`
- `supporting_artifact_ids`
- `related_graph_proposal_ids`
- `related_label_ids`
- `metrics`
- `recommendation`
- `builder_version`
- `created_at`
- `updated_at`

Suggested statuses:

- `active`
- `reviewed`
- `dismissed`
- `archived`
- `superseded`

The MVP can either:

1. create a new table such as `syncratic_insights`; or
2. write synthesized rows into `insight_ledger` with metadata that marks them as Syncratic multi-event insights.

Recommended MVP: create new Syncratic-specific tables first. Do not overload `insight_ledger` until the migration path from one-signal insights is explicit.

## Insight Creation Rules

G088 should be deterministic and conservative.

Initial grouping rules:

- group by tenant, use case, subject symbol, signal type family, and fixed time window;
- require at least two supporting signals or alerts unless explicitly configured as a single-event explanatory insight;
- prefer alerts as the operational evidence boundary for actionable signals;
- include lower-severity signals only as supporting context, not as alert-like incidents;
- compute summary metrics from persisted evidence only;
- avoid natural-language generation beyond deterministic templates.

Initial strategies:

- `symbol_signal_cluster_5d`: same symbol, related DSM signal family, last 5 trading sessions;
- `symbol_recurrence_20d`: same symbol and signal type recurring over 20 trading sessions;
- `graph_proposal_context`: alerts/signals that support the same graph fact key or proposal subject;
- `promotion_evidence_context`: candidate/evaluation/baseline evidence supporting a promotion review.

## Selective Materialization

G088 should not create a Syncratic batch job, context window, or insight for every asset simply because the MarketOps Top 50 universe exists.

The recommended pattern is:

```text
daily Top 50 scan -> candidate counts -> threshold-gated context build -> materialized insight
```

The daily scan should be cheap and aggregate-first. It can inspect all Top 50 assets after EOD processing, but it should only fetch full evidence and persist Syncratic rows for candidates that cross configured thresholds.

MVP materialization rules:

- scan the Top 50 MarketOps asset universe daily after EOD signal/alert persistence;
- group candidate counts by tenant, use case, context strategy, subject symbol, signal family, and strategy window;
- materialize a context window only when at least one default trigger is met:
  - two or more related alerts or signals exist in the strategy window;
  - one or more critical alerts exist in the strategy window;
  - related graph proposal, evaluation, promotion, or operator-label evidence exists for the same subject/window;
- skip materialization when the existing context window has the same evidence digest;
- create at most one active synthesized insight for the same tenant, use case, strategy, subject symbol, window, and builder version;
- enforce configurable run caps such as max assets scanned, max candidate windows, max context windows materialized, and max insights materialized;
- record run metrics for scanned assets, candidate windows, materialized contexts, materialized insights, unchanged skips, below-threshold skips, and budget-cap skips.

The deterministic idempotency key should include:

- `tenant_id`
- `use_case`
- `context_strategy`
- `subject_symbol`
- `window_start`
- `window_end`
- `context_builder_version`

On-demand UI actions may request a preview for a quiet asset, but preview-only output should not become the canonical Syncratic insight unless it is persisted through the same threshold, digest, idempotency, and audit path.

## Alert Versus Insight Semantics

Alerts:

- event-level;
- created from medium/high/critical signals;
- direct operational response;
- may map directly to an incident/case;
- lifecycle: `open`, `acknowledged`, `resolved`, `suppressed`.

Syncratic insights:

- multi-event or multi-record pattern;
- generated from context windows;
- explanation and synthesis;
- references supporting alerts/signals/events;
- lifecycle: `active`, `reviewed`, `dismissed`, `archived`, `superseded`.

G088 should not immediately delete existing one-signal insights. It should introduce the new model and document a later migration/suppression plan.

## APIs

Suggested context window APIs:

- `POST /v1/syncratic/context-windows`
- `GET /v1/syncratic/context-windows`
- `GET /v1/syncratic/context-windows/{context_window_id}`

Suggested synthesized insight APIs:

- `POST /v1/syncratic/insights`
- `GET /v1/syncratic/insights`
- `GET /v1/syncratic/insights/{syncratic_insight_id}`
- `POST /v1/syncratic/insights/{syncratic_insight_id}/review`
- `POST /v1/syncratic/insights/{syncratic_insight_id}/dismiss`
- `POST /v1/syncratic/insights/{syncratic_insight_id}/archive`

Create context request example:

```json
{
  "tenant_id": "tenant-local",
  "subject_symbol": "AAPL",
  "context_strategy": "symbol_signal_cluster_5d",
  "window_start": "2026-07-07T00:00:00Z",
  "window_end": "2026-07-12T00:00:00Z",
  "signal_types": ["marketops.dsm.volatility_expansion"]
}
```

Create insight request example:

```json
{
  "tenant_id": "tenant-local",
  "context_window_id": "synctx-...",
  "insight_type": "marketops.syncratic.recurring_volatility_cluster"
}
```

## UI Direction

Frontend should eventually make the distinction obvious:

- Alerts page: incident/work queue, event-level actions, urgency and response lifecycle.
- Syncratic Insights page: pattern library, context windows, supporting evidence, explanation, and review lifecycle.

Do not simply clone the Alerts UI for Syncratic Insights.

A future frontend spec should include:

- context window detail panel;
- supporting alert/signal/event references;
- evidence digest and builder version;
- explanation summary;
- related graph proposals and operator labels;
- review/dismiss/archive controls.

## Acceptance Criteria

G088 is complete when:

- context windows can be deterministically created from existing ledgers;
- synthesized insights can reference context windows and supporting evidence;
- insight creation requires multi-event evidence for the default strategies;
- Top 50 daily coverage uses aggregate candidate scanning and does not eagerly materialize quiet assets;
- unchanged evidence digests are skipped rather than rewritten;
- materialization caps prevent excessive Syncratic context/insight creation in one run;
- alerts remain event-level and unchanged;
- one-signal insight duplication is documented and not expanded;
- APIs expose context windows and synthesized insights;
- tests cover deterministic ids, grouping rules, minimum evidence thresholds, unchanged digest skips, materialization caps, idempotent reruns, and tenant isolation;
- authenticated API smoke validates create/list/detail for context windows and synthesized insights.
